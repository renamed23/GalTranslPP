**翻译整体流程概览**

- 前端（CLI / GUI）只需要提供：  
  - 一个项目目录 `projectDir`（内含 `config.toml` 与 `gt_input` 等）。  
  - 一个实现了 `IController` 的控制器实例（负责进度条和日志显示）。  
- 工作流（由库内部完成）：  
  1. `createTranslator(projectDir, controller)` 读取配置并创建合适的 Translator 实例；  
  2. Translator 在 `run()` 中依次调用 `init()` → `beforeRun()` → `process()` → `afterRun()`；  
  3. 每个文件被解析成若干 `Sentence`，经过字典/插件的预处理，分批送入 LLM 或脚本翻译，再经后处理与问题分析器检查；  
  4. 整个过程中的日志与进度都通过 `IController` 回调传回前端。

      - 处理关闭事件，配合 `UpdateChecker` 在必要时触发更新安装。  

- `BasePage.*`  
  - 所有设置页面的基类，封装共有的标题、布局和“应用配置”接口 `apply2Config()`。

- `ProjectSettingsPage.*`  
  - 每个具体项目的设置页，包含一个 `QStackedWidget`，组合多个子设置页：  
    - `APISettingsPage`：选择/配置翻译 API（model 名、baseURL、策略等）。  
    - `PluginSettingsPage`：选择文本插件（Lua/Python/C++）和执行时机。  
    - `CommonSettingsPage`：通用翻译参数（线程数、批大小、排序方式、目标语言等）。  
    - `PASettingsPage`：问题分析（ProblemAnalyzer）相关设置。  
    - `NameTableSettingsPage`：姓名表配置。  
    - `DictSettingsPage` / `DictExSettingsPage`：项目级字典与扩展字典设置。  
    - `StartSettingsPage`：翻译启动页（包含“开始翻译”按钮），通过 `TranslatorWorker` 启动后台翻译线程。  
    - `OtherSettingsPage`：其它项目级参数。  
    - `PromptSettingsPage`：当前项目提示词配置（通常覆盖默认 `Prompt.toml`）。  
  - 负责把页面上的控件值写回到项目的 `config.toml` 中，并发出 `finishTranslatingSignal` 和 `changeProjectNameSignal` 等通知。

- `SettingPage.*`  
  - 全局设置页，绑定到 `globalConfig`（例如 Python 环境路径、语言、自动更新配置等），并通过信号通知主窗口重启 Python 环境。

- 字典与姓名表相关模型：  
  - `GptDictModel.*`：`QAbstractTableModel` 的实现，用于 GUI 中显示/编辑 GPT 字典条目（与核心库 `GptDictionary` 概念对应，但类型独立）。  
  - `NormalDictModel.*`：普通字典模型。  
  - `NameTableModel.*`：项目姓名表的模型。  
  - 这些模型被各种 `*DictPage`/`NameTableSettingsPage` 使用。

- 翻译执行与进度显示：  
  - `TranslatorWorker.*`：  
    - 继承自 `QObject`，封装一个后台线程任务槽 `doTranslation()`：  
      - 内部创建 `GUIController`（`TranslatorWorker.cpp` 中定义的 `IController` 实现），将其传给 `createTranslator`。  
      - 将核心库中产生的日志和进度通过 Qt 信号转发给 GUI（`writeLogSignal` / `makeBarSignal` / `updateBarSignal` 等）。  
      - 支持 `stopTranslation()`，通过 `IController::shouldStop()` 查阅 `_shouldStop` 标志，在翻译逻辑中优雅中断。  
  - `GUIController`：  
    - 与 CLI 的 `TerminalController` 类似，但通过 Qt 信号将数据推送到 UI，而不是直接打印到控制台。  
    - 自带 flush 线程，周期性合并/截断日志并刷新进度。

- 更新相关：  
  - `UpdateChecker.*`：  
    - 使用 `QNetworkAccessManager` 访问 GitHub Releases API，解析 `tag_name` 并调用 `cmpVer` 判断是否有新版本。  
    - 支持自动下载 GUI 核心包 `GUICORE.7z`，校验 digest（SHA-256），提示托盘通知，并在应用关闭后由 `Updater` 完成替换。  
    - 与 `globalConfig` 获取 `autoDownloadUpdate` 等配置。

- 其它 GUI 组件（简要）：  
  - `HomePage`：首页/欢迎页。  
  - 各种 `*Page`：对应不同配置区域的子页面（Epub/PDF 配置、插值参数、跳过规则、换行修复配置等）。  
  - UI 控件类：`ElaDoubleText`、`ElaInputDialog`、`PluginItemWidget`、`UpdateWidget`、`ValueSliderWidget`、`NoWheelComboBox` 等封装常用控件行为。  
  - `Resource/` 与 `GPPGUI.qrc`：Qt 资源文件，包含图标、翻译文件、样式等。

**更新器 `Updater`**

- 命令行参数：  
  - `--pid`：主程序进程 ID。  
  - `--source`：更新包路径（通常是 `.7z`）。  
  - `--target`：安装目标目录。  
  - `--restart`：更新后要重新启动的主程序路径。  
  - `--newActionFlag` 及若干 `--*Version`：新版本与当前版本对比的辅助信息。  
- 流程：  
  - 在“普通更新模式”下：  
    - 等待主程序退出 → 解压 `Updater_new.exe` 和必要 DLL 至 `target/new` → 启动新的 Updater 实例并立即返回。  
  - 在 `--newActionFlag` 模式下：  
    - 再次等待主程序退出，解析各种版本号，调用 `cmpVer` 判断是否兼容/是否需要覆盖 Prompt/字典等。  
    - 计算需要排除的前缀（如用户的字典/脚本路径），调用 `extractZipExclude` 完成解压替换。  
    - 如果指定 `--restart`，在更新完成后重新启动主程序。
